# case-filing-gen 起诉状/答辩状生成

## 工作定位

基于S9裁判预测和S6争点矩阵，生成起诉状或答辩状。必须律师确认要点后才生成。

**触发条件**：S10门禁通过后，律师确认
**必须确认**：**是**（确认后才生成）
**门禁**：必须先通过 S10 blocking_result.is_blocked=false 且 can_enter_final=true，方可生成诉讼文书。S10 未执行视同门禁失败。

---

## 执行流程

### 第一步：门禁校验

调用 `check_final_gate.py` 校验 S10 blocking_result：
- `is_blocked=false` 且 `can_enter_final=true` → 通过，继续第二步
- 否则 → **CRITICAL阻断**，不生成诉讼文书

### 第二步：读取九步法文件

- 读取 `intermediate/原告九步法/S1-固定权利请求/`
- 读取 `intermediate/原告九步法/S9-要件归入与裁判预测/`
- 读取 `intermediate/原告九步法/S6-争点矩阵/`

### 第三步：列出要点清单

基于S9裁判预测，列出起诉状/答辩状的要点清单：
1. 诉讼请求
2. 事实与理由
3. 法律依据
4. 证据清单

### 第四步：律师确认

展示要点清单："以下是起诉状要点，请确认是否可以生成"

**红线**：律师确认后才生成。

**重要**（2026-06-09新增）：起草任何法律文书前，必须先确认律师的思路和结构要求，不得按旧框架直接执行。接到"重写""修改"指令时，先回顾之前的讨论记录，确认律师的具体要求再动手。

### 第五步：生成文书

律师确认后，调用 `generate_filing_materials.py` 生成。

### 第六步：修改追问机制

律师修改时，系统必须追问修改原因：
- 诉讼请求写法不同 → 记住律师习惯，下次按此格式
- 诉讼策略变更 → 必须展开讨论，分析利弊
- 质量问题 → 记录律师的改法，沉淀为写作经验

### 第七步：调用Hook

```bash
~/.claude/skills/case-os/scripts/case-post-step.sh [案件路径] filing-gen
```

---

## 红线

- ❌ **S10 门禁不通过不得生成**
- ❌ **律师确认后才生成**
- ❌ 不得绕过 `check_final_gate.py` 直接调用生成脚本
- ❌ 修改时必须追问原因
