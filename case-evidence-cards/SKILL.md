# case-evidence-cards（A5）证据卡片与关系复核

## 工作定位

从 `_archive/markdown/` 目录中的材料提取结构化证据卡片，生成双视图输出（按来源+按目的），供后续九步法使用。

**前置条件**：A4（案件基础信息与案情概要）确认
**必须确认**：**是**（律师复核，含证据卡片 + 关系推断）
**执行后**：调用Hook脚本更新状态

---

## 执行流程

### 第一步：读取材料

读取 `_archive/markdown/` 目录下所有Markdown文件。

### 第二步：提取证据卡片

逐份材料提取结构化证据卡片：

| 字段 | 说明 |
|------|------|
| 编号 | SP001, SP002... |
| 来源文件 | 原始材料文件名 |
| 原文摘要 | 关键内容摘录 |
| 发言者 | 己方/对方/第三方/不确定（聊天记录必须标注） |
| 证明目的 | 承认欠款/确认事实/时间节点/金额记录/身份关系/合同约定/其他 |
| 时间 | 相关日期（如有） |
| 金额 | 相关金额（如有） |
| 置信度 | 高/中/低（低置信度标记"待确认"） |

### 第三步：发言者识别规则（聊天记录专用）

对于聊天记录类材料（微信、短信、QQ等），每条证据卡片**必须标注发言者**：
- **己方**：我方当事人说的话
- **对方**：对方当事人说的话
- **第三方**：群聊中其他人的发言
- **不确定**：无法判断发言者身份

判断依据：
1. 聊天记录中通常有发言者昵称/备注名
2. 结合CLAUDE.md中的当事人信息判断
3. 无法判断时标"不确定"，标记⚠️待律师确认

**法律意义**：对方说的话=对方自认（强证据）；己方说的话=己方主张（需佐证）

### 第四步：双视图输出

| 视图 | 文件名 | 分组方式 | 用途 |
|------|--------|----------|------|
| **来源视图** | `证据卡片库-按来源.md/json` | 按来源材料文件分组 | 材料溯源、文件级审阅 |
| **目的视图** | `证据卡片库-按目的.md/json` | 按证明目的标签分组 | 九步法分析、要件映射 |

**来源视图格式**：
```markdown
## 来源：微信聊天记录.md
| 编号 | 原文摘要 | 发言者 | 证明目的 | 时间 | 金额 | 置信度 |
|------|----------|--------|----------|------|------|--------|
| SP001 | "我确实欠你16万" | 对方 | 承认欠款 | 2026-02-05 | ¥161,300 | 高 |
| SP002 | "你赶紧还钱" | 己方 | 催收事实 | 2026-02-10 | | 中 |
```

### 第五步：证据关系推断（历史 A7.5 能力，纳入 A5）

律师复核通过后，运行关系推断：

```bash
python3 ~/.claude/skills/case-os/scripts/infer_relations.py "[案件根目录]" --action scan
```

输出文件：
- `intermediate/S0-关系推断提示词.md` — AI推断提示词

**AI推断**：将上述提示词交给 AI，AI 依据以下规则推断证据间关系：

| 关系类型 | 识别方式 | 精确定位要求 |
|---------|---------|------------|
| **时间线关联** | 相同/相近日期或时间段 | 标注到具体段落的日期表述 |
| **印证关联** | 共享实体（金额、人名、合同号、事件表述） | 标注两处证据中相互印证的确切语句 |
| **矛盾关联** | 同一事实的不同/冲突描述 | 标注矛盾双方的具体原文段落 |
| **链条关联** | 时间先后+因果逻辑 | 标注链中每一步的节点和切换依据 |

AI推断完成后，将输出保存到文件，然后解析入库：

```bash
python3 ~/.claude/skills/case-os/scripts/infer_relations.py "[案件根目录]" --action parse --ai-output /path/to/ai_output.md
```

输出文件：
- `intermediate/S0-证据关系.json` — 结构化关系数据

然后生成复核清单：

```bash
python3 ~/.claude/skills/case-os/scripts/infer_relations.py "[案件根目录]" --action review
```

输出文件：
- `intermediate/S0-证据关系复核.md` — 关系复核清单

### 第六步：律师复核（含关系复核）

**复核方式**（两种都支持）：
1. **Markdown复核**：直接打开复核清单逐条确认
2. **对话端**：主人说"关系1确认"、"关系3驳回"等

**复核内容**：
#### 证据卡卡片复核（原有）
- 证明对象是否正确
- 文字摘要是否有错误
- **发言者是否标注正确**（聊天记录类必须逐条核对）
- 编号是否连贯

#### 证据关系复核（新增）
复核并确认后，更新 S0-证据关系.json 中的 status 字段：

```bash
# 确认一条关系：更新 status 为 "confirmed"
# 驳回一条关系：更新 status 为 "rejected"
# 修改一条关系：更新 status 为 "modified"，修改具体字段，填写 review_note
```

**关系复核要点**：
1. **关系是否真正成立**：不要疏漏重要的关联，也不要接受牵强的关联
2. **精确定位是否准确**：锚点是否精准到了正确段落
3. **三层可见性是否合理**：律师工作台(全量)、客户简报(关键)、法官摘要(核心)
4. **矛盾关联是否遗漏**：仔细检查是否有被忽略的证据之间的矛盾

**证据卡片复核完成后**，调用 update_evidence_status 更新 S0-证据关系.json 的 meta 确认状态：

```json
"meta": {
  "confirmed": true,
  "confirmed_at": "2026-05-18T10T",
  "confirmed_by": "律师",
  ...
}
```

### 第七步：定稿

律师复核确认后，生成最终文件集合：
- `intermediate/S0-证据卡片库.md` — 证据卡片（已有）
- `intermediate/S0-证据卡片库.json` — 证据卡片结构化数据（已有）
- `intermediate/S0-证据关系.json` — 证据关系数据（新增）
- `intermediate/S0-证据关系复核.md` — 关系复核存档（新增）

### 第八步：证据门户（历史 A7.8 展示能力，默认停用）

证据门户为 legacy 可视化资产，不是 A5 正常交付、确认入口或发布数据源。
仅在律师明确要求查看可视化页面后，才可将门户 HTML 复制到案件根目录并打开：

```bash
# 部署证据门户（自动注入案件数据）
python3 ~/.claude/skills/case-os/scripts/deploy_portal.py "[案件根目录]"

# 产物：案件根目录/evidence-portal.html
# 脚本自动打开浏览器
```

**门户功能**：
- **总览页**：统计卡片（证据数/关系数/置信度）、关系分布柱状图、三层视图可见性统计
- **图谱页**：vis-network 交互式关系网络图（节点=证据卡，边=关系，虚线=矛盾），支持按关系类型筛选
- **时间线页**：按时间排序的证据时间线，颜色区分发言者
- **证据链页**：因果关系链可视化展示，按链条分组
- **数据表页**：可搜索/筛选/排序的证据数据表格，点击行查看详情
- **三层切换**：右上角按钮切换 律师/客户/法官 视图

**数据加载**：
- 门户自动查找 `intermediate/S0-证据卡片库.json` 和 `intermediate/S0-证据关系.json`
- 文件不存在时自动回退到内置演示数据

### 第九步：写入 A5 确认并自动完成 A6 内部门禁

律师明确复核完成后，先形成 A5 确认 JSON 记录并执行：

```bash

  "[案件路径]" --step A5 --confirmation-file "[确认记录JSON路径]"
```

```bash

```

```

✅ A6 内部门禁已由状态生成器自动校验并写回；不得要求律师另行执行 A6。
📌 下一步：A6 状态通过时，直接进入 Phase B 的 S1。
```
