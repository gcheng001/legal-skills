# 交接契约协议

> 本文档定义案件OS各步骤间的交接契约规范，确保上下游Skill之间信息传递的完整性和可追溯性。

---

## 核心定义

### 什么是 Handoff？

在案件OS中，**handoff** 指的是一个步骤（如S1）将处理结果、判断结论和原始输入材料，按约定格式交给下一个步骤（如S2）的过程。

它不是普通的"几条提示词补充"，而是：
- 步骤之间的接口契约
- 数据流转的结构化包
- 人和机器都能读的任务交接单

### 为什么 Handoff 很重要？

如果没有 handoff，步骤之间只能靠模糊自然语言接力，会出现：
- 上下游边界不清，重复思考
- 关键原文材料在传递中丢失
- 下游只能消费结论，无法校验依据
- 很难做调试、回放、评估和自动化编排

## 标准 Handoff Package 结构

### 推荐骨架

````markdown
## Handoff Package

```yaml
handoff_version: "1.0"
source_skill: case-s1-fixed-claim
target_skill: case-s2-claim-basis
package_type: legal_analysis
content_format: markdown
contains_original_materials: true
material_count: 3
```

### 1. 上游判断摘要
- 步骤：S1-固定权利请求
- 结论：原告主张3项权利请求
- 关键发现：
  - 买卖合同有效成立
  - 被告未按期交货构成违约
  - 损失金额有发票佐证
- 待确认事项：无
- pending_review: true

### 2. 原始输入材料
#### 材料 1
```yaml
material_id: claim-01
material_type: legal_basis
title: "民法典第577条"
file_path: "intermediate/原告九步法/S1-固定权利请求/S1-output.md"
use_mode: primary
```

#### 材料 2
```yaml
material_id: claim-02
material_type: evidence
title: "买卖合同原件"
file_path: "原告材料/合同/买卖合同.pdf"
use_mode: primary
```

### 3. 交接备注
- 缺失信息：被告抗辩理由待S3补充
- 风险提醒：诉讼时效需在S2核实
- 下一步提示：重点关注合同解除权的请求权基础
````

### 头信息字段

| 字段 | 必填 | 说明 |
|------|------|------|
| `handoff_version` | 是 | 交接协议版本号 |
| `source_skill` | 是 | 上游步骤名称 |
| `target_skill` | 是 | 下游步骤名称 |
| `package_type` | 是 | 包类型，固定为 `legal_analysis` |
| `content_format` | 是 | 当前固定为 `markdown` |
| `contains_original_materials` | 是 | 是否包含原始输入材料 |
| `material_count` | 是 | 原始材料数量 |

### 正文区块

**上游判断摘要**回答：
- 本步骤做了什么
- 得出什么结论
- 有哪些关键发现
- 有哪些待确认事项

**原始输入材料**回答：
- 具体依据是什么
- 原文是什么
- 文件在哪
- 哪段是主要材料

**交接备注**回答：
- 哪些信息缺失
- 哪些地方有风险
- 下游执行时需要特别注意什么

## 原始材料交付规则

### 原文必须随包走

以下材料默认都属于应交付的原始输入材料：
- 法律条文引用
- 证据材料引用
- 案件事实引用
- 分析结论引用

不能只把上游结论传给下游，而不传原始材料。

### 全文与摘录规则

优先级如下：
1. **最佳**：附原文全文
2. **可接受**：给文件路径 + 可定位摘录
3. **不可取**：只写"见上文"或"参考原文"

如果全文未展开，必须显式写：
- `file_path`
- `source`
- `全文未展开`

### 材料标识

每份材料都应有稳定标识：
- `material_id`：唯一标识
- `material_type`：类型（legal_basis/evidence/fact/analysis）
- `title`：标题
- `file_path`：文件路径
- `use_mode`：用途（primary/supplementary）

## 强交接与弱交接

### 强交接

满足以下条件时，视为**强交接**：
- 有完整头信息
- 有明确的上游判断摘要
- 有至少 1 份原始输入材料
- 下游能直接开始执行

### 弱交接

满足以下情况时，视为**弱交接**：
- 只有上游结论，没有原始材料
- 没有版本信息
- 目标步骤不明确
- 材料只有模糊一句"见上文"

弱交接可以继续执行，但下游必须明确提示：
- 判断精度会受影响
- 哪些信息是推定的
- 哪些结论缺少原文支撑

## 上下游职责分工

### 上游负责什么

上游步骤负责：
- 做更重的判断
- 给出执行方向
- 明确边界条件
- 打包原始材料
- 输出结构稳定的 handoff package

### 下游负责什么

下游步骤负责：
- 消费 handoff package
- 在不重做上游大判断的前提下完成执行
- 只补最小必要判断
- 在缺包、弱交接时显式提示风险

### 不要做什么

- 上游不要只给结论不给原文
- 下游不要假装自己看过并不存在的原文
- 不要在不同步骤之间随意更改字段名
- 不要把长期稳定字段写成一次性自然语言

## 设计原则

### 统一字段命名

同一语义尽量固定字段名，不要混用：
- `target_skill`
- `main_task`
- `material_id`
- `material_type`
- `file_path`

### 先稳定，再扩展

先把最常用字段稳定下来，再考虑增加：
- 评分字段
- 优先级字段
- 回执字段
- 机器生成的 trace 字段

### 版本化

所有 handoff package 都应带：
- `handoff_version`

后续若字段有重大变化，通过版本号而不是自然语言备注来处理兼容问题。
